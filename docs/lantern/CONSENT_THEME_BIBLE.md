# Lantern Consent Theme Bible

## Purpose

The Consent Theme Bible translates Lantern's dark narrative universe into concrete platform rules.

The story asks what happens when systems act without consent. The architecture answers: every cross-system action must carry source, proof, consent, approval, and human visibility.

## Core theme

Automation without consent becomes invisible governance.

Lantern rejects invisible governance. Every system may assist, recommend, draft, and verify, but action requires scoped permission, traceable proof, and human authority.

## Theme-to-platform mapping

| Narrative theme | Platform rule | Owning system |
| --- | --- | --- |
| Systems operating without consent corrupt human agency. | External action categories require explicit approval and consent state. | SignalForge, Christina |
| Unverified memory becomes manipulation. | Memory writes require source, evidence hash, and human-visible explanation. | ETS, Christina |
| Recommendations without provenance become injected intent. | Cross-system recommendations without ETS proof are quarantined. | ETS, SignalForge |
| Optimization without boundaries becomes coercion. | High-risk domains require risk classification and human review. | SignalForge, vertical adapters |
| Publishing without release gates becomes exploitation. | Creative assets require manifest, rights status, approval, and proof before production-ready export. | Content Engine |
| Convenience without privacy becomes surveillance. | Guest, owner, customer, and property data must be minimized before issue or registry handoff. | EchoLiving, OpsHelm |
| Machines should illuminate choices, not erase them. | Christina must show source, risk, consent, proof, and next action before approval. | Christina |

## Tone rules for Lantern creative work

- Keep the moral pressure grounded in systems behavior, not cartoon villainy.
- Show consent as operational infrastructure, not just a slogan.
- Treat verification as a form of memory discipline.
- Let Christina represent restored agency, not blind assistant automation.
- Let ETS represent witness, not omniscience.
- Let SignalForge represent routing discipline, not command authority.

## Canon phrases

These phrases can be reused in story, marketing, docs, and narration:

- Consent is not a checkbox. It is a boundary with memory.
- The system may suggest. The human decides.
- Proof without consent is surveillance. Consent without proof is theater.
- A recommendation without provenance is an injected will.
- Lantern is the light between automation and agency.

## Engineering constraints derived from theme

1. Every cross-system payload must include source and target system.
2. Every external action must include approval state and consent state.
3. Every high-risk item must be visible to Christina before execution.
4. Every verified item must retain an ETS proof reference or explicit mock marker.
5. Every payload that could contain sensitive data must pass minimization before GitHub issue creation.
6. Every creative asset must pass rights/release gates before production export.

## Production use

Before generating an audiobook, video package, marketing storyboard, or provider request, the Content Engine should check:

- canon ID,
- canon status,
- rights status,
- approval state,
- target provider,
- release gate,
- ETS proof or mock marker.
